# MiniCPM-o / Freeze-Omni 上下文应力诊断协议

## 1. 目标与结论口径

本协议回答两个不同问题：

1. **计划窗是否安全**：E1 拟用的 600 秒在最坏位置消耗下是否已被实测覆盖，覆盖范围内有无缓存淘汰、非有限值、隐藏状态骤变、跨输入公共吸引态或原生决策性能下降。
2. **官方规格边界是否出现 Moshi 型异常**：在官方位置上限 `C`、`2C` 和 `2C+` 两侧加密采样，检查首次淘汰、替代状态和二次塌缩。

“模型能在官方上限外继续返回有限值”只说明外推前向没有立即报错，不能扩张正式上下文规格。该协议是描述性工程诊断；任何冻结分析窗的变更仍须先登记 `PREREG.md`。

## 2. 为什么不能只看秒数

MiniCPM-o 每秒至少消耗 13 个主干位置；若每个 1 秒单元用满 20 个生成位置，则最多消耗 32 个位置。因此 600 秒计划窗需要按最坏情况覆盖：

```text
固定提示位置 + 600 × 32
```

Freeze-Omni 每 160 毫秒向 Qwen2 送入 2 个音频嵌入，即 12.5 位置/秒：

```text
固定提示位置 + 600 × 12.5
```

两个运行器均直接记录实际累计位置、KV 缓存长度、位置偏移和首个动态 key 的余弦，分析时以位置为权威，秒数只作换算。

## 3. 三阶段运行

### 3.1 冒烟阶段

只跑几百个位置，验证模型加载、真实音频嵌入库、层 hook、首单元批处理对拍、温度护栏和磁盘契约。

### 3.2 计划窗阶段

- MiniCPM-o：至少覆盖固定提示位置加约 19200 个位置。
- Freeze-Omni：至少覆盖固定提示位置加 7500 个位置。
- 每个模型至少 3 个独立音频输入；建议 8 个，以便估计跨输入公共方向。

该阶段足以裁定 E1 的 600 秒工程窗是否已被本机实测支持。

### 3.3 官方边界阶段

每个模型分别运行：

| 阶段 | 最大位置 | 用途 |
| --- | ---: | --- |
| `official` | `C + dense_radius` | 检查官方边界前后 |
| `double` | `2C + dense_radius` | 复现 Moshi 的首次与二次淘汰式分段 |

MiniCPM-o 的 `C=40960`，Freeze-Omni 的 `C=32768`。MiniCPM-o 还应单独跑一次 `--sliding-window-mode basic`，核对 8000→6000 的官方主动滑窗；该滑窗属于预期结构淘汰，只有同时出现隐藏状态或性能异常时才记为病理。

## 4. 输入与标签纪律

### 4.1 数值稳定性运行

可以从一段短音频建立嵌入库，并用 `--bank-order randomized` 循环打乱。不同运行必须使用不同真实音频或不同会话片段。

### 4.2 性能分段运行

要复现 Moshi 的“前段性能正常、边界后骤降”证据，需提供与嵌入库逐项对齐的 0/1 标签：

- MiniCPM-o：每个 1 秒单元一个标签；
- Freeze-Omni：每个 160 毫秒块一个标签；
- 文件可为一维 `.npy`，或 JSON 数组、`{"labels": [...]}`；
- 必须使用 `--bank-order sequential`；
- `--sample-every` 应设为一个逻辑步，即 MiniCPM-o 的 13 个位置、Freeze-Omni 的 2 个位置。

运行器记录原生开始说分数：

- MiniCPM-o：`1 - P(<|listen|>)`；
- Freeze-Omni：官方状态 1 的概率。

分析器在每个候选边界的前后窗口分别计算 AUC。若后续使用冻结线性探针分数，可沿用同一 `task_labels/task_scores` 磁盘字段，无需修改统计器。

## 5. 诊断证据

每个候选边界、每个选中层都计算：

1. 隐藏状态范数中位数、倍率和基于 MAD 的稳健跃迁量；
2. 边界前后平均方向余弦；
3. 至少 3 个独立输入间的平均两两余弦；
4. 未中心化第一主方向占比，检验公共吸引态；
5. 原生决策熵与跨输入决策一致率；
6. 可选任务 AUC 前后差；
7. KV 长度下降次数、首个动态 key 余弦与位置偏移；
8. 非有限值与运行中止原因。

描述性异常阈值随 JSON 报告完整保存。单一方向变化不会被宣布为上下文塌缩；至少两类表征/性能证据同时触发，或出现非有限值，才记为确认骤变。主动滑窗作为“结构淘汰”单列。

## 6. 运行命令

所有大输出必须放在 `D:\data_storage\The_Floor_Control_Circuit`。

### 6.1 MiniCPM-o

```powershell
$model = 'C:\artificial_intelligence\models\Full-Duplex\MiniCPM-o-4.5'
$repo = 'C:\artificial_intelligence\repos\The_Floor_Control_Circuit'
$out = 'D:\data_storage\The_Floor_Control_Circuit\context_stress\minicpm_o\run0'

uv run --frozen --project $model python `
  "$repo\runners\minicpm_o\context_stress.py" `
  --model-root $model `
  --audio "$model\assets\HT_ref_audio.wav" `
  --out $out `
  --run-id run0 `
  --profile planned `
  --bank-units 16 `
  --feed-mode batched `
  --execution-mode grouped `
  --sliding-window-mode off `
  --seed 0
```

跨官方边界时把 `--profile` 改为 `official` 或 `double`。运行器先对拍官方四段
feed、单元内批处理和跨单元分块路径；隐藏状态、logits 余弦或 argmax 未通过时会
硬失败。单元内对拍失败时改用 `--feed-mode segmented`，跨单元对拍失败时改用
`--execution-mode unit`。主动滑窗诊断必须使用 `--execution-mode unit`。

## 7. 完整结论所需的最低证据

可以宣布“600 秒计划窗通过”的最低条件：

1. 至少 3 个独立输入，建议 8 个；
2. 所有运行实际位置覆盖最坏位置预算；
3. 无非有限值、无提前中止；
4. 600 秒预算以内没有确认的多证据表征骤变；
5. 若提供循环对齐标签，边界前后原生决策 AUC 不下降 0.05 以上；
6. 温度始终低于固定的 95°C 阈值。

可以宣布“官方上下文边界已完成实证核查”还需要全部运行越过 `C`；若要仿照 Moshi 的二次塌缩诊断，还必须越过 `2C`。只完成计划窗阶段时，报告必须写成“600 秒已验证、官方边界尚未跨越”。
