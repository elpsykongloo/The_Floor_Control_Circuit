# E1 探针网格加速分析

日期：2026-07-22

代码基线：`03df5f4`（`main` 与 `origin/main` 一致）

## 结论

原双进程按层奇偶分片方案超出本机物理内存，且仓库根环境原先安装的是 CPU 版 torch，按旧命令无法启动正式 CUDA 网格。本轮将执行链改为单进程共享层缓存、双 GPU 按规格并行，并消除训练、评估、标签读取、bootstrap 与格子写盘中的重复工作。科学会话、标签、采样、C 网格、优化目标、概率值与 G2 判据保持不变。

正式 parity 与全网格仍需先完成 Zarr 摄取和 339 个缺失会话的标签增算；本轮没有把缺少前置产物的状态描述成正式网格已跑通。

## 本机约束

- GPU：2 × RTX 3090，每卡 24 GiB。
- 物理内存：31.75 GiB。
- 单层 1000 路激活：`1000 × 3000 × 4096 × fp16 = 22.89 GiB`。
- 旧双进程方案仅层缓存副本就需要 45.78 GiB；旧训练路径还会叠加特征类型转换副本，必然触发换页。
- 新方案按层先载入训练侧 800 路（18.31 GiB），拟合完成后释放，再载入评估侧 200 路（4.58 GiB）。
- 根环境已锁定 Windows 官方 CUDA 12.8 wheel；实查 `torch 2.11.0+cu128`、CUDA 12.8、两张设备可见。

## 实测基准

| 项目 | 原路径 | 新路径 | 结果 |
|---|---:|---:|---:|
| 20 万行二分类 AUC | 0.05547 s | 0.01498 s | 3.70× |
| 20,000 × 1,024、五档 C | 0.7028 s | 0.2535 s | 2.77× |
| 两张 GPU、各 50,000 × 1,024 | 1.0422 s 串行 | 0.5374 s 并行 | 1.94× |
| 6 个 10 万行格子、1000 次会话 bootstrap | — | 2.726 s | 完成 |
| 100 万行格子写盘 | 压缩 0.3519 s | 直接 0.0425 s | 8.28× |
| 100 万行格子读取 | 压缩 0.0641 s | 直接 0.0348 s | 1.84× |

格子代表样本的未压缩体积为 9.63 MiB，压缩后为 7.10 MiB；压缩节省 26%，正式网格预计多占约 1–2 GiB 数据盘。逐会话键、dtype、数组值、原子替换与 `np.load` 接口兼容。

## 真实数据盘点

- 修正生产标签契约后，冻结的 500 个 E1 会话中有 339 个缺标签。
- 其余 161 个会话的完成标记、标签 SHA-256、源音频大小/修改时间及登记过的历史代码/设置指纹均有效。
- 161 个已有会话的行域：每个 T1 δ 规格 757,433 行，T2 12,413 行，T3 9,310 行，T4 9,223 行，T5 步长抽样后 241,500 行。
- 10 规格并行标签读取在 3 个工作线程下耗时 10.735 s；旧重复读取路径运行 180 s 仍未完成。

## 已落地的加速

1. 单进程持有一份主存层缓存，两张 GPU 由固定线程按估算行数贪心分配规格。
2. 特征按角色块从 fp16 缓存直接写入 GPU f32 张量；总体均值/方差用 Chan 合并公式计算。
3. 同格五档 C 复用标准化后的设备张量并沿 C 路径热启动；inner 评分共享一次特征搬运。
4. 同规格三种子共享评估装配和设备搬运；训练后先写小型拟合断点，训练/评估间中断可跳过 18.31 GiB 重读。
5. 每个标签 parquet 对全部规格只读一次，行域持久化为带配置与清单签名的 NPZ。
6. 会话 bootstrap 使用正负有序对充分统计量与计数二次型；测试逐次结果与字面重采样一致。
7. 有效秩九个 k 共用一次最大 rank=128 投影。
8. 格子 NPZ 停用低收益压缩，保留完整断点兼容性。

## 标签生成候选探索

事件标签最大显式前瞻为 2.0 s。用 244 s 音频前缀生成标签，并与完整会话在正式分析步范围内比较，10 个真实会话全部逐行一致，总耗时 42.95 s。话轮合并存在跨边界链式延伸的理论可能，因此该方案暂未接入正式产物链；启用前应完成已有 161 个会话的全量逐行校验，并为前缀产物建立独立指纹。

## 新执行入口

```powershell
uv sync

$dataRoot = 'D:\data_storage\The_Floor_Control_Circuit'
$env:FLOOR_CIRCUIT_IO_JOBS = '3'
uv run python scripts/wp_e1_ingest.py `
  --plan "$dataRoot\e1_cache_plan\e1_r1_moshi.plan.json"

uv run python scripts/wp_e1_probe_grid.py --stage labels
uv run python scripts/wp_e1_run_missing_events.py `
  --session-list "$dataRoot\e1_probe\missing_label_sessions.txt"

uv run python scripts/wp_e1_probe_grid.py --stage labels
uv run python scripts/wp_e1_probe_grid.py --stage parity --device cuda

# 在四个 pwsh 窗口分别运行，并把 shard-id 设为 0、1、2、3。
uv run python scripts/wp_e1_probe_grid.py --stage acoustic `
  --num-shards 4 --shard-id 0

uv run python scripts/wp_e1_probe_grid.py --stage run `
  --num-shards 1 --devices cuda:0,cuda:1

uv run python scripts/wp_e1_probe_grid.py --stage finalize --device cuda
```

31.75 GiB 本机上不要再开两个 `--num-shards 2` 窗口；内存护栏会在读取层缓存前拒绝这一启动方式。
